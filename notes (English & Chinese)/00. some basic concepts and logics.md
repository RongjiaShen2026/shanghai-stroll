# 目录结构的选择

```
shanghai-stroll/
├── src/
│   ├── __init__.py            ← 空文件，让 src 变成 Python 包
│   ├── dijkstra.py            ← 你现有的算法（Node/Edge/Graph/Dijkstra）
│   └── graph_adapter.py       ← 新建：osmnx → WeightedDigraph 的转换
├── notebooks/
│   └── explore.ipynb          ← 你现在的 Notebook
└── README.md
```


## __init__.py

Python 看到一个文件夹里有 __init__.py，就把这个文件夹当成"包"，允许你写 from 文件夹名.文件名 import 东西。没有它，Python 不认这个文件夹是 Python 代码的家。

它就是个标记文件，告诉 Python："这个文件夹是个包，请把里面的 .py 文件当成模块来对待。" 内容可以完全为空。没有它，from src.dijkstra import ... 在某些 Python 版本和工具（特别是 Streamlit、pytest）里会报"找不到 src"。加上它，万无一失。

### 用 Python 包（带 __init__.py 的文件夹）有什么好处？

1. 让 import 路径清晰、可追溯

没有包结构时，所有 .py 文件平铺在根目录：

```
shanghai-stroll/
├── dijkstra.py
├── graph_adapter.py
├── data_loader.py
├── visualizer.py
├── poi_finder.py
├── app.py
└── explore.ipynb
```

文件一多就乱套——分不清哪些是"我自己写的代码"，哪些是"配置文件、脚本、Notebook"。
有了包：

```
shanghai-stroll/
├── src/                ← 一眼看出：这里是我的源代码
│   ├── __init__.py
│   ├── dijkstra.py
│   ├── graph_adapter.py
│   └── ...
├── explore.ipynb       ← 探索性脚本
├── README.md           ← 文档
└── requirements.txt    ← 配置
```

src/ 是"代码的家"，根目录是"项目的家"，职责分明。


2. 避免命名冲突

如果你写了一个 utils.py，Python 安装的某个第三方库也有 utils.py，平铺结构下 import utils 会撞车。from src.utils import xxx 就不会——src.utils 是独一无二的"完整路径"。

3. 让代码可以被任何地方调用

包结构后，无论你的 Notebook 在哪个文件夹、Streamlit 的 app.py 在哪个文件夹，都能用同一行 from src.dijkstra import Dijkstra 调用你的算法。代码搬家不会破坏 import。


4. 这是 Python 社区的通用约定

打开任何开源项目（pandas、requests、Django），都是 src/包名/ 或者 包名/ 这种结构。用包结构 = 你的项目"看起来像 Python 项目"，别人能立刻读懂。


# 知识点 2
```python
from src.dijkstra import Dijkstra
```
import 这个动作，Python 会把整个 dijkstra.py 从头到尾跑一遍——包括测试代码。

**Python 的 import 不是"挑出某个东西"，而是"先把整个文件跑一遍，然后给你你点名要的那个名字"。**

from dijkstra import Dijkstra 在 Python 看来 = 两步：

1. 执行整个 dijkstra.py（一行一行从头跑到尾）
2. 跑完后，从生成的"命名空间"里把 Dijkstra 这个名字交给你

```python
import heapq                   # ← import 时执行：会真的去加载 heapq 库

class Node: ...                # ← import 时执行：创建 Node 这个类对象

def Dijkstra(...): ...         # ← import 时执行：创建 Dijkstra 这个函数对象

# 下面这些也会被执行 ↓↓↓
g = WeightedDigraph()          # ← 真的会创建一个图
g.addNode(Node('A'))           # ← 真的会加节点
dist, path = Dijkstra(g, ...)  # ← 真的会跑算法
print(f'shortest distance...')  # ← 真的会打印
```

你写 from dijkstra import Dijkstra，Python 怎么知道 Dijkstra 是个函数？它必须先执行到第 40 行那个 def Dijkstra(...):，这个 def 语句执行后，Dijkstra 这个名字才在内存里存在。

**换句话说：def、class 本身也是要执行的语句。它们不是"声明"，是"动作"——执行的时候才会把函数/类对象创建出来。**

所以为了找到 Dijkstra，Python 必须跑完整个文件（至少跑到那一行）。它做不到"只跑某一部分"——因为它事先不知道 Dijkstra 在哪一行，也不知道这个函数依赖哪些前置定义。

### 文件里"光秃秃、不在函数/类里"的代码，叫模块级代码。这些代码在 import 时会跟着 def 和 class 一起被执行。

Python 没办法分辨"哪些是必要的定义、哪些是不希望 import 时跑的测试"——它一律执行。
**if __name__ == '__main__'**: 就是你亲手告诉 Python "下面这段只在直接运行时跑"的方式。Python 没有读心术，必须你显式标记。


## import的原则

一个简单的原则

只 import 你真的会用到的东西。

不必要的 import 有几个坏处：

1. 误导读者——别人看到 from src.dijkstra import Dijkstra 会以为这个文件里跑了算法，结果搜半天找不到调用
2. 耦合更紧——以后如果 Dijkstra 改名或拆分，这个文件无端被牵连
3. 代码风格不专业——Linter（代码检查工具）会标黄"未使用的 import"

关键事实

当 Python 加载 src/dijkstra.py 时（任何 import 都会触发这个加载），文件从头跑到尾，所以整个文件被加载之后，Edge、Node、Digraph 这些类都已经在 src.dijkstra 这个模块的命名空间里存在了。

比如，from src.dijkstra import WeightedEdge 只是说："把 WeightedEdge 这个名字搬到我的文件里来"。它没有切断 WeightedEdge 和 Edge 之间的继承关系——那个关系是在 src.dijkstra 模块里建立好的，永远在那里。

**一句通用法则：你 import 的名字 = 这个文件里会直接写出来的名字。**



# 知识点 3
```python
if __name__ == '__main__'
```

if __name__ == '__main__': 是一个开关——它让一段代码只在你"直接运行这个文件"时执行，而在"被别人 import"时不执行。


## __name__ 是什么？

每个 Python 文件被加载时，Python 会给它一个内置变量 __name__。这个变量的值取决于它是怎么被加载的：

|加载方式|__name__ 的值|
|---|---|
|python src/dijkstra.py（直接运行）|'__main__'|
|from src.dijkstra import ...（被别人 import）|'src.dijkstra'（也就是它的模块名）|

所以这个判断的逻辑是：

```python
if __name__ == '__main__':
    # 翻译：如果"我是被直接运行的"，那就执行下面的测试
    ...
```
**直接运行 → 跑测试，确认算法没坏。**

**被 import → 跳过测试，安静地把类和函数交出去。**



# 知识点 4：加速积累的Tips

1. 方法 1：每写完一段代码，回头写一行"它在干什么"

不是注释代码"怎么做"，而是用大白话写它"在解决什么问题"。

比如你这段：
```python
for osmid in G.nodes: 
    node = Node(str(osmid))
    wg.addNode(node)
    osmid_to_node[osmid] = node
```
写一行总结：

"这段把 NetworkX 的整数 id 节点，转成我自己 Node 类的字符串 id 节点，同时记下对应关系，方便以后转回去。"

久了你会发现：很多代码段其实是同一种模式的变体。


2. 给"我做过的事"起一个名字

你今天写的这个，叫"适配器"或"翻译层"。

明天你可能会写：

1. 预处理：清洗数据让它能用
2. 包装：在原对象外面加一层方便调用
3. 过滤：从大数据里筛出需要的
4. 聚合：把多份数据合成一份

每次给自己写的代码起个类别名字。慢慢你脑子里就有一张地图。

3. 方法 3：先画图，再写代码

下次写转换代码前，先在纸上画两边的数据结构：

```
INPUT: G (NetworkX MultiDiGraph)
   ├─ G.nodes: NodeView of int
   │    └─ 53017091, 53017092, ...
   └─ G.edges: with 'length' attr
        └─ (u, v, {length: 45.2})

OUTPUT: wg (WeightedDigraph)  
   ├─ nodes: Node(str_id)
   └─ edges: weight=length

MAPPING NEEDED:
   int (osmid)  ←→  Node object
   (用 osmid_to_node 这个 dict 来存)
```

把两边的"形状"画出来，差距一目了然，要写什么代码也一目了然。

资深程序员在脑子里做这件事，你现在画在纸上做——本质上是一样的。

# 1. How to Structure Your Project Folder

```
shanghai-stroll/
├── src/
│   ├── __init__.py     ← empty folder，let src be a Python package
│   ├── dijkstra.py     ← Your algorithm（Node/Edge/Graph/Dijkstra）
│   └── graph_adapter.py  ← new script:osmnx → WeightedDigraph 
├── notebooks/
│   └── explore.ipynb          ← Notebook to explore codes
└── README.md
```


## __init__.py

What __init__.py specifically does

It's a marker file that tells Python: "This folder is a package — treat the .py files inside as modules." It can be completely empty.

Without it, from src.dijkstra import ... will fail with "src not found" in certain Python versions and tools (especially Streamlit and pytest). With it, you're safe.


### What's the benefit of using a Python package (a folder with __init__.py)?

1. Clear, traceable import paths

Without a package structure, all .py files are flat in the root directory:

```
shanghai-stroll/
├── dijkstra.py
├── graph_adapter.py
├── data_loader.py
├── visualizer.py
├── poi_finder.py
├── app.py
└── explore.ipynb
```

This gets messy fast — you can't tell which files are "my source code" and which are "configs, scripts, notebooks."

With a package:

```
shanghai-stroll/
├── src/                ← clearly: this is my source code
│   ├── __init__.py
│   ├── dijkstra.py
│   ├── graph_adapter.py
│   └── ...
├── explore.ipynb       ← exploration script
├── README.md           ← docs
└── requirements.txt    ← config
```

src/ is the "home for code"; 

the root is the "home for the project." 

Clear separation of concerns.


2. Avoid naming collisions

If you write a utils.py and some installed third-party library also has a utils.py, a flat structure makes import utils ambiguous. from src.utils import xxx won't collide — src.utils is a unique "full path."

3. Code is callable from anywhere

With a package structure, no matter where your Notebook lives or where Streamlit's app.py lives, you can use the same from src.dijkstra import Dijkstra to call your algorithm. Moving files around doesn't break imports.

4. It's the Python community convention

Open any open-source project (pandas, requests, Django) and you'll see src/package_name/ or package_name/ structures. Using a package = your project "looks like a Python project" and others can read it immediately.


# 2. Explaination 2:

```python
from src.dijkstra import Dijkstra
```

Python's import isn't "pick something out of a box" — it's "execute the entire file, then hand you the name you asked for."

from dijkstra import Dijkstra is two steps:
1. Run all of dijkstra.py top to bottom
2. From the resulting namespace, hand you the name Dijkstra

```python
import heapq                   # ← import 时执行：会真的去加载 heapq 库

class Node: ...                # ← import 时执行：创建 Node 这个类对象

def Dijkstra(...): ...         # ← import 时执行：创建 Dijkstra 这个函数对象

# 下面这些也会被执行 ↓↓↓
g = WeightedDigraph()          # ← 真的会创建一个图
g.addNode(Node('A'))           # ← 真的会加节点
dist, path = Dijkstra(g, ...)  # ← 真的会跑算法
print(f'shortest distance...')  # ← 真的会打印
```

### Why must it work this way?
def and class aren't declarations — they're statements that execute to create function/class objects. Until Python reaches def Dijkstra(...): and executes it, the name Dijkstra doesn't exist in memory.

So to find Dijkstra, Python has to run the whole file (at least up to that line). It can't "only run part of it" because it doesn't know in advance where Dijkstra is defined or what it depends on.


### What about "naked" code in the file?
Top-level code (not inside a function or class) is called module-level code. It runs during import alongside the defs and classes. Python can't distinguish "necessary definitions" from "test code I didn't want to run on import" — it runs everything.

if __name__ == '__main__': is how you explicitly tell Python "skip this block when imported." Python can't read minds — you have to mark it.


### The principle of importing?

When Python loads src/dijkstra.py (which any import triggers), the whole file runs top to bottom. By the time it's done, Edge, WeightedEdge, and the inheritance relationship between them all live inside the src.dijkstra module's namespace. They're fully linked.

from src.dijkstra import WeightedEdge just brings the name WeightedEdge into your file. It doesn't sever the inheritance — that link was built inside src.dijkstra and stays there forever.

You only need to import the parent class if your file directly references its name (e.g., isinstance(x, Edge)).

Universal rule

**What you import = what you literally write in this file.**





# 3. Explaination 3:

### 1. What does if __name__ == '__main__': do?

It's a switch that makes a block of code run only when you execute the file directly, not when the file is imported by someone else.

What goes wrong without it?
When you import a Python file, Python runs the entire file top to bottom — including any test code at the bottom. So if your dijkstra.py has a print statement at the end, importing it from a Notebook will randomly print things you didn't ask for.


### 2. What is __name__?

Every Python file gets a built-in variable __name__ when loaded:

|How it's loaded|Value of __name__|
|---|---|
|python src/dijkstra.py (run directly)|'__main__'|
|from src.dijkstra import ... (imported)|'src.dijkstra' (the module name)|

So the idiom reads as: "If I'm being run directly, run the test block below. Otherwise, skip it."

### 3. Analogy

Think of dijkstra.py as a toolbox. Running it directly = opening it and testing the wrench works. Importing it = someone borrowing a wrench — they don't need to watch you test every tool.

### 4. Why it matters in real project

1. Notebook imports stay clean (no surprise output)
2. Streamlit's app.py won't trigger weird side effects when it imports
3. The test code stays available — just run python src/dijkstra.py anytime to verify the algorithm still works. Cheap self-test.
