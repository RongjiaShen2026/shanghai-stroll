
# How to create project and install packages through Terminal

## A. basic command
### 1. 找一个你喜欢的位置,创建项目文件夹
```bash
mkdir shanghai-stroll
cd shanghai-stroll
```

### 2. 创建 conda 环境(名字叫 shanghai-stroll,用 Python 3.11)
```bash
conda create -n shanghai-stroll python=3.11 -y
```

### 3. 激活环境(激活后命令行前面会出现 (shanghai-stroll))
```bash
conda activate shanghai-stroll
```

### 4. 装包-——用 conda-forge 这个 channel 装地理库最稳
```bash
conda install -c conda-forge osmnx folium matplotlib jupyter -y
```

### 5. 装完验证：
```bash
python -c "import osmnx, networkx, folium, matplotlib; print('osmnx:', osmnx.__version__); print('networkx:', networkx.__version__); print('folium:', folium.__version__); print('✅ 全部就位')"
```

### 6. 验证状态：
1. 看 conda 环境列表(看看 shanghai-stroll 有没有建过)
```bash
conda env list
```

2. 看项目文件夹位置
```bash
ls ~/Desktop/Projects/shanghai-stroll
```

### 7. 日常做项目：进入 + 激活
1. 进入 + 激活
```bash
cd ~/Desktop/Projects/shanghai-stroll
conda activate shanghai-stroll
```

2. 用 VS Code 打开
```bash
code .
```

如果 code 命令报错,在 VS Code 里:

         1. 按 Cmd+Shift+P
         2. 输入 Shell Command: Install 'code' command in PATH
         3. 回车,完成后重启终端


### 注意:一定要在terminal里打开VS Code：

code . —— 这是从已经激活了 shanghai-stroll 环境的终端里启动 VS Code。

## 这一步暗藏了一个关键机制：子进程继承父进程的环境。

核心机制：环境会"继承"

当 Terminal.app 里激活了 shanghai-stroll，然后你敲 code . 启动 VS Code：

```python
Terminal.app（环境：shanghai-stroll）
         │
         │ 启动 VS Code
         ▼
       VS Code（继承了 shanghai-stroll 环境）
         │
         ├── Notebook → 继承 → shanghai-stroll
         ├── 内置终端 → 继承 → shanghai-stroll
         └── 按▶️运行 .py → 继承 → shanghai-stroll
```

VS Code 整体都会"继承"启动它的那个终端的环境。

但如果你是从别处启动 VS Code（比如点 Dock 图标、Spotlight 搜索打开），它就不继承任何环境——默认走 base。

所以，不难发现，当点击VS Code图标进入VS Code时，打开VS Code的终端，会发现还在base环境中：

```
(base) rongjiashen@RongjiadeMacBook-Pro shanghai-stroll % 
```


### 补充：如果找不到文件或者没在正确路径下安装项目文件：
1. 先找到 shanghai-stroll 现在在哪里

```bash
find ~ -type d -name "shanghai-stroll" 2>/dev/null
```

2. 在桌面创建Projects文件夹 + 搬家
```bash
mkdir -p ~/Desktop/Projects

mv /Users/你的名字/shanghai-stroll ~/Desktop/Projects/

cd ~/Desktop/Projects/shanghai-stroll

pwd

ls
```

| 命令 | 作用 |
|---|---|
| `mkdir -p ~/Desktop/Projects` | 在桌面创建 Projects 文件夹。`-p` 表示"如果已经存在就不报错" |
| `mv 旧路径 新路径` | **m**o**v**e,把文件夹搬到新位置(像剪切粘贴) |
| `cd ~/Desktop/Projects/shanghai-stroll` | 进入搬过去之后的位置 |
| `pwd` | 确认现在在哪 |
| `ls` | 列出里面的文件 |

## C. Explanation：
1. 
```bash
mkdir shanghai-stroll  
```
创建一个名叫 shanghai-stroll 的文件夹：mkdir = make directory(创建文件夹)

2. 
```bash
cd shanghai-stroll     
```

进入这个文件夹：
         
         cd = change directory(切换目录 / 进入文件夹)

执行后,你的终端"当前位置"就变成了 shanghai-stroll 文件夹里面。之后你在终端里干的所有事(创建文件、装包、跑代码),默认都发生在这个文件夹里。
             
3. conda create -n shanghai-stroll python=3.11 -y

* -n 是 conda 的一个参数(flag),告诉 conda:"接下来这个词是环境的名字"

* -y 自动同意所有提示

不加 -y 会发生什么? conda 会在执行前列出它要装的所有包,然后停下来问你: 你得手动敲 y 然后回车,它才继续。加了 -y, conda 默认 "当作你已经回答了 yes",直接开装,不问你。

* python=3.11
这是 conda 命令里的一个参数,意思是 "给这个新环境装 Python 3.11"。
如果你不写 python=3.11,conda 会用一个默认版本(通常是它自己 base 环境的版本)。但我强烈建议你显式指定,原因有两个:
       1. 不同 Python 版本兼容性不同
       2. 要部署到 Streamlit Cloud。Streamlit Cloud 会问你 "用哪个 Python 版本部署?"——如果你本地用的 3.11,云上也指定 3.11,就不会出现"本地能跑、云上崩"的灵异事件。

这个命令不用先查"我系统里有没有 3.11"——conda 会自己去下载Python 3.11,装到这个新环境里,跟你电脑上原本的 Python 完全不冲突。这正是 conda 的优势之一:每个环境的 Python 版本都可以不一样,互不干扰。

* 注意：conda 环境不放在项目文件夹里,而是统一存在 conda 自己的目录(~/anaconda3/envs/ 或类似路径)。

想看准确位置,激活后跑: 
               
                 conda info --envs
会列出所有环境和它们的真实路径。

记住：环境位置 ≠ 代码位置。它们是两件独立的事情:
conda create -n shanghai-stroll 的 -n 参数告诉 conda:"在你统一的仓库里给我新建一个叫这个名字的环境"—— -n 就是 name(名字)的意思,跟"在哪个文件夹执行命令"无关。
你在任何文件夹下执行 conda create -n shanghai-stroll python=3.11 -y,结果都一样,环境永远建在 conda 自己的仓库里。
项目和环境靠激活环境的动作绑定在一起,而不是两者的位置。

* 环境和项目最好用同名，因为你 cd 进项目文件夹,看到提示符里 (shanghai-stroll),立刻知道"环境对上了"。



4. 
```bash
conda install -c conda-forge osmnx folium matplotlib jupyter -y
```

装包优先用 conda install -c conda-forge 包名,装不到的再用 pip install


5. 
```bash
python -c "import osmnx, networkx, folium, matplotlib; print('osmnx:', osmnx.__version__); print('networkx:', networkx.__version__); print('folium:', folium.__version__); print('✅ 全部就位')"
```

这是在不打开 Python 解释器的情况下,用一行命令快速验证几个包是否装好。
它做了 4 件事:

       1. 启动 Python
       2. 导入 osmnx、networkx、folium、matplotlib 这 4 个包
       3. 打印它们的版本号
       4. 如果一切顺利,打印 "✅ 全部就位"
       5. 如果任何一个包没装好,Python 会立刻报错 ModuleNotFoundError,你就知道哪个出了问题。

-c
这是 python 命令的一个参数,意思是 command —— "接下来引号里的字符串,当作 Python 代码直接运行"。


6. 
```bash
find ~ -type d -name "shanghai-stroll" 2>/dev/null
```

find = 查找命令

~ = 从你的用户主目录开始找(覆盖 Desktop、Documents、Downloads 等几乎所有地方)

-type d = 只找文件夹(directory),不找文件

-name "shanghai-stroll" = 名字叫这个的

2>/dev/null = 把"权限不足"那些烦人的报错隐藏掉

如果啥都没输出(命令跑完直接回到提示符),说明文件夹真的没创建——这种情况我下面也会处理。








## D. VS Code, Jupyter and Markdown

### D1. VS Code and Jupyter
VS Code已经内置支持 Jupyter,你打开一个 .ipynb 文件,VS Code 就自动变成 Jupyter界面了。

VS Code(Visual Studio Code) 是一个编辑器——你可以用它写任何代码:Python、JS、HTML、Markdown……它就像 Word,但是为程序员设计的。

Jupyter不是编辑器,是一种运行代码的方式。它的特点是: 把代码分成一个个小格子(cell)，一个一个跑,跑完立刻看到结果，结果(数字、图表、地图)就显示在每段代码下面。

打个比方:
1. 写 .py 文件 = 写一篇完整的文章,然后一口气从头读到尾
2. 写 .ipynb(Jupyter)文件 = 写实验笔记本,每做一个实验记一段,边做边看结果

在VS Code里：创建第一个 Notebook
在 VS Code 里:
   1. 点 File → New File
   2. 文件名写 explore.ipynb(注意后缀是 .ipynb,不是 .py)
   3. 保存到 shanghai-stroll 文件夹里

### 让 Notebook用你的虚拟环境，这一步很关键,否则你装的包它找不到。
### 打开 .ipynb 文件后,右上角会有一个**"Select Kernel"**按钮(或者显示当前用的Python版本):
### 点它，选 "Python Environments..."找到你的 .venv(应该会显示路径里有 shanghai-stroll/.venv),选它

在探索阶段为什么用Notebook更好？
因为，就拿osmnx输入地图来说，如果你只是想改一下画图的样式,在 .py 里你得每次都重新下载地图,而在 Notebook 里,一个cell跑过一次后变量就在内存里了,你只需要反复跑下一个cell。对探索性工作来说,这是天壤之别。


### D2. Markdown
.md 文件本质上就是 .txt——纯文本。这意味着:
     
        1. 任何编辑器都能打开(VS Code、记事本、TextEdit……)
        2. 不会出现"这个版本的 Word 打不开旧文档"的破事
        3. 文件超小(几 KB)
        4. 永远不会"格式损坏"

学会 Markdown,这些场景全都通用:
   
        1. GitHub 的 README(你以后项目主页就用 Markdown 写)
        2. 技术博客(掘金、Medium、知乎都支持)
        3. Jupyter Notebook 的笔记 cell
        4. Notion、Obsidian、飞书文档(基本都支持 Markdown 语法)
        5. 聊天工具:Slack、Discord、ChatGPT 输入框,都吃这套符号, Claude 也是用 Markdown 写回复的。
        6. Stack Overflow 的问答

此外，Markdown能版本管理，因为是纯文本,你写的笔记可以用 git 追踪修改历史。Word 文档做不到这一点(Word 是二进制文件,git 看不懂改了什么)。

### 在VS Code种边写边预览： 
###                    Mac:Cmd + K,松开,再按 V 
###                    Windows:Ctrl + K,松开,再按 V

### Markdown我还不熟的语法：

1. 加粗           **加粗**           加粗

2. 斜体            *斜体*            斜体

3. 列表            - 项目            • 项目

4. 行内代码         `代码`             代码

5. 代码块          三个反引号包起来

```python
print("Hello World")
```

常用的"语言标签"：跟在三个反引号后面写,告诉 Markdown 这是什么语言:

| 标签 | 用于什么 |
|---|---|
| bash 或 shell | 终端命令 |
| python | Python 代码 |
| javascript 或 js | JS 代码 |
| html | HTML |


拆解：
| 标签 | 用于什么 |           ← 第 1 行:表头(列名)
|---|---|                       ← 第 2 行:分隔线(必须有!)
| bash 或 shell | 终端命令 |    ← 第 3 行起:数据
| python | Python 代码 |

1. 竖线 | = 列分隔符

每一列之间用 | 隔开。行首和行尾的 | 可加可不加(加上更整齐)。

2. 第二行的 --- = 必须有的分隔线

这一行告诉 Markdown:"上面是表头,下面是数据"。没有它 → Markdown 不认为这是表格,会显示成普通文字
至少要 3 个减号 ---(2 个不行)

3. 每一行 = 一行数据

数据按列对齐,每列用 | 分隔。

4. 进阶:对齐方式

在分隔线上加冒号 :,可以控制每列的对齐:

| 左对齐 | 居中 | 右对齐 |
|:---|:---:|---:|
| 文字 | 文字 | 文字 |
| 长一点的文字 | 长一点的文字 | 长一点的文字 |


## E. 为什么需要虚拟环境？
1. 如果我们不用虚拟环境，今天做new project 并且装了一个包，比如osmnx 1.9，下一个月需要维护一个老project，它要求 osmnx 0.16(API 完全不一样)，当我执行 pip install osmnx==0.16,把新版本覆盖了,那么new project立刻坏掉,而且会报错信息。更糟的是:osmnx 还依赖 numpy、pandas、shapely……一个包升级,可能把十几个其他包的版本顶飞,引发连环崩盘。这种情况有个外号叫 "dependency hell"(依赖地狱)。

2. 用虚拟环境后,requirements.txt 干净
部署到 Streamlit Cloud(阶段 B)时,你需要列出项目用到的所有包。如果你直接用系统Python,跑一下 pip freeze 会列出你电脑上装的所有Python包——包括跟这个项目无关的几十个。用虚拟环境后,pip freeze 只列出这个项目真正用到的包,部署时干净利落。

3. 实际应用中：
在terminal：

(base) rongjiashen@RongjiadeMacBook-Pro shanghai-stroll %

(shanghai-stroll) rongjiashen@RongjiadeMacBook-Pro shanghai-stroll %

(...) 里的名字 = 你现在激活的 conda 环境。

(shanghai-stroll) ... ~ %   ← 当前激活的是 shanghai-stroll 环境

(base) ............ ~ %     ← 当前激活的是 base 环境

两者的 ~(位置)是一样的,变的是"环境",不是"位置"。

你不需要"在 shanghai-stroll 文件夹里"才能使用 shanghai-stroll 环境。只要终端提示符开头是 (shanghai-stroll),无论你在哪个文件夹,Python 都会用 shanghai-stroll 环境里的包。

base 是 conda 安装时自带的默认环境。它的存在是为了: 让 conda 本身有个家(conda 命令自己也要 Python 来跑)，给你一个"啥也没装"的干净起点

### 最佳实践:不要往 base 里装包,每个项目用自己的环境。base 越干净越好。
### 在不同环境下，新建文件夹，效果是一样的。因为 mkdir 和 mv(创建文件夹、移动文件夹)是系统命令,它们由 macOS 系统本身提供,跟 Python、跟 conda 没有任何关系。conda 环境只影响"Python 相关的事",不影响"操作文件系统的事"。


## 环境只在"运行"那一刻起作用。"打开"、"编辑"、"保存"都和环境无关。


## 进一步理解environment：

比喻：环境 = 工作台上的工具组合

想象你家有两个房间，每个房间都有一张工作桌：

1. 房间 A：电脑自带的 Terminal.app
2. 房间 B：VS Code 里的内置终端

每张桌子上都可以摆不同的工具组合：

1. "base 工具组" = 通用工具（默认就在桌上）
2. "shanghai-stroll 工具组" = 你为这个项目准备的工具（osmnx、networkx、folium 等）

关键事实：

1. 每张桌子是独立的——你在房间 A 的桌上摆了"shanghai-stroll 工具组"，完全不影响房间 B 的桌子
2. 房间 B 的桌子默认摆的是"base 工具组"——你没动它，它就一直是这样
3. conda activate shanghai-stroll 这个命令 = "把这张桌子上的工具换成 shanghai-stroll 工具组"——只对你敲命令的那张桌子有效

所以：

1. 你在 Terminal.app（房间 A）敲了 conda activate shanghai-stroll → A 桌换上了项目工具
2. VS Code 终端（房间 B）你没敲过 → B 桌还是默认的 base 工具

两张桌子各管各的。没有"全局激活"这种东西——激活是针对单个 shell session 的。

### Notebook 不在任何一个"房间的桌子"上。它有自己专属的"工作站"，叫 kernel。

VS Code 让你在 Notebook 右上角直接选这个工作站用哪套工具：

```python
shanghai-stroll (Python 3.11.15)   ← 你之前点过的
```

### 你点了这个，就告诉 Notebook："你的工作站用 shanghai-stroll 工具组"。从此以后 Notebook 里跑代码，用的就是这套工具——完全不管两个终端的状态。

这就是为什么你之前 VS Code 终端是 (base)，但 Notebook 跑代码完全没事——Notebook 走的是 kernel 那条独立路线。

实操规则

记住这三条：
1. 每个终端窗口/标签都是独立的——激活只对当前这一个有效
2. 想知道当前激活了什么：看提示符开头的 (xxx)
3. Notebook 用的是 kernel：在 VS Code 右上角选过的那个，跟终端无关
