
# Exception

Exception 是 Python 里几乎所有错误的父类。当你想自己定义一个错误类型时，继承它，就自动获得了"被 raise、被 try/except 捕获、打印出 traceback"的全部能力。


## 异常的继承family tree

Python 的异常不是散乱的，它们组成一棵继承树：

```
BaseException                  ← 最顶层（一般不直接碰）
 ├── SystemExit                ← sys.exit() 触发的，不算"错误"
 ├── KeyboardInterrupt         ← 你按 Ctrl+C
 └── Exception                 ← ★ 你日常遇到的所有错误都在这下面
      ├── ValueError           ← 值不对（比如 int('abc')）
      ├── KeyError             ← 字典里没这个 key
      ├── TypeError            ← 类型不对（比如 1 + 'a'）
      ├── FileNotFoundError    ← 文件找不到
      ├── ZeroDivisionError    ← 除以 0
      └── ...（还有几十个）
```

注意 Exception 和 BaseException 的区别：你自定义异常时继承 Exception 而不是 BaseException。因为 Exception: 会捕获所有普通错误，但不会误捕 Ctrl+C 和 sys.exit()——这是设计上故意的，让用户永远能中断程序。


## Exception 的核心行为不是用纯 Python 写的，而是用 C 写在 CPython 解释器里的。

### 1.  第一层：用 Python 能看到的部分
```python
print(Exception.__mro__)        # 它的继承链
print(dir(Exception))           # 它有哪些方法和属性
```

__mro__（Method Resolution Order）会输出：

```
(<class 'Exception'>, <class 'BaseException'>, <class 'object'>)
```

这告诉你：Exception 继承自 BaseException，BaseException 继承自 object（万物之祖）。真正的核心能力在 BaseException 里。

dir(Exception) 你会看到 args、__traceback__、with_traceback、__str__ 等等——这些就是它对外暴露的能力。


### 2. 第二层：被 raise、被 try/except 捕获、打印出 traceback, 这些能力从哪里来？

1. 为什么能被 raise？

raise 不是随便什么对象都能用。CPython 解释器在执行 raise X 时，会检查 X 是不是 BaseException 的子类（或实例）。这个检查是写死在解释器里的规则。

```python
raise 42          # TypeError: exceptions must derive from BaseException
raise Exception() # ✅ 通过检查
```

所以"能被 raise"不是 Exception 类里有个叫 raise 的方法，而是解释器只认 BaseException 家族。继承它 = 拿到这张准入证。


2. 为什么能被 try/except 捕获？
except SomeError: 的匹配逻辑，本质是解释器做一次 isinstance(抛出的异常, SomeError) 判断：

```python
try:
    raise ValueError("oops")
except Exception as e:        # ValueError 是 Exception 的子类 → isinstance 为真 → 捕获
    print("caught:", e)
```
输出是：
```python
caught: oops
```

```
进入 try
↓
执行 raise ValueError("oops")
↓
程序发现出错了
↓
去 except 里找能不能处理这个错误
↓
except Exception 可以处理 ValueError
↓
把错误存到变量 e 里
↓
打印 caught: oops
```
这段代码的意思是：
我故意制造了一个 ValueError，然后用 except Exception as e 把它捕获，并把错误信息打印出来。


3. 为什么能打印 traceback？

这分两半：

1. traceback 数据是解释器生成的，不是异常对象自己造的。
当异常向上"冒泡"穿过一层层函数调用时，CPython 在每一层把"当前在哪个文件、哪一行、哪个函数"记录下来，挂到异常对象的 __traceback__ 属性上。异常对象只是个"容器"，记录工作是解释器做的。

2. 打印成那串文字，是 __str__ 和 traceback 模块负责的。
异常未被捕获、冒泡到顶层时，解释器调用 traceback 模块把 __traceback__ 里的信息格式化成你熟悉的那串：

```python
Traceback (most recent call last):
  File "app.py", line 10, in <module>
    ...
ValueError: oops
```

最后那行 ValueError: oops 里的 oops，来自异常对象的 args——super().__init__("oops") 存进去的就是它。


# 实际应用

```python
class GraphConversionError(Exception):
    """Raised when a NetworkX graph cannot be converted to a WeightedDigraph."""
```

三种写法：

```python
# 写法 1: 只有 docstring,不加 pass (推荐)
class GraphConversionError(Exception):
    """Raised when a NetworkX graph cannot be converted to a WeightedDigraph."""


# 写法 2: 加 pass
class GraphConversionError(Exception):
    """Raised when a NetworkX graph cannot be converted to a WeightedDigraph."""
    pass


# 写法 3: 只有 pass,没有 docstring
class GraphConversionError(Exception):
    pass
```
**三种都能跑、都没语法错误。推荐写法 1。**

Python 的规则

Python 不允许"空"的代码块——class、def、if、for 后面必须有内容，否则会报 IndentationError。

```python
class Foo:        # ❌ SyntaxError
                  # 后面什么都没有

```
### 所以 Python 提供两种"占位"方式：

### 方式 A：用 pass

```python
class Foo:
    pass          # pass 是个"什么都不做"的占位语句
```

### 方式 B：用 docstring
```python
class Foo:
    """A class that does nothing."""
                  # docstring 本身就算一条语句!
```
**关键点：docstring 在语法上就是一条合法的语句——它会被存进 Foo.__doc__。所以有了 docstring，class 体就不空了，不需要再加 pass。**

docstring 的特殊待遇：当字符串字面量出现在 class/function 的第一条语句位置时，Python 会自动把它保存为 __doc__ 属性——所以它既"占位"又"有用"。

## 工业代码里自定义异常的标准写法：
```python
class GraphConversionError(Exception):
    """Raised when a NetworkX graph cannot be converted to a WeightedDigraph."""
```
就这样——一行 docstring，到此结束。不加 pass。

这是 Python 主流项目（Django、Flask、Pandas、Requests）里都遵循的风格。

### 异常机制为什么这样设计。我分两部分讲：(1) 为什么需要自定义异常，(2) 为什么它必须是 class 而不是函数

**关键事实：Python 的异常机制是建立在 class 之上的**

在 Python 里，异常本质上是"对象"，而对象来自 class。raise 和 except 这套机制，底层完全依赖类和继承。

1. raise 在做什么？

```python
raise GraphConversionError("图是空的")
```
这一行做了两件事：

1. GraphConversionError("图是空的") —— 创建一个该类的实例（对象）
2. raise —— 把这个对象"抛"出去

所以异常必须是个能被实例化的东西——也就是 class。函数没法这样用。


2. except 靠什么匹配?

```python
except GraphConversionError as e:
    ...
```
意思是：

如果前面的 try 代码块里发生了 GraphConversionError 这种错误，就进入这里处理，并且把这个错误对象命名为 e。

Python 捕获异常时，用的是 isinstance 检查——"抛出来的这个对象，是不是 GraphConversionError 类（或其子类）的实例？"

```python
# Python 内部大致是这么判断的:
if isinstance(抛出的异常对象, GraphConversionError):
    # 匹配成功,进入这个 except 块
```
**isinstance 检查的就是"类的归属关系"——这是 class 系统才有的能力。函数没有"实例""继承""isinstance"这些概念，根本没法参与这套机制。**


## 继承是异常分类的核心
前面提到的"错误家族"：

```python
class GraphConversionError(Exception):
    ...

class EmptyGraphError(GraphConversionError):  # 继承
    ...
```

为什么 except GraphConversionError 能抓住 EmptyGraphError？

因为 EmptyGraphError 是 GraphConversionError 的子类，isinstance(空图错误对象, GraphConversionError) 返回 True。

这套"父类抓子类"的能力，完全依赖 class 的继承机制。函数没有继承，做不到。

## 为什么必须继承 Exception？

```python
class GraphConversionError(Exception):
    #                       ^^^^^^^^^ 必须继承 Exception (或其子类)
```

# Python 规定：只有继承自 BaseException 的类，才能被 raise。

```python
class NotAnException:   # 没有继承 Exception
    pass

raise NotAnException()  # ❌ TypeError: exceptions must derive from BaseException
```
所以你的异常类必须继承 Exception——这是"接入异常系统"的入场券。Exception 这个父类已经实现了异常该有的所有底层行为（存储错误消息、生成 traceback 等），你继承它就白白获得这些能力。







```python
try:
    wg, _ = nx_to_weighted_digraph(G)
except GraphConversionError:
    # 转换问题：可能换个数据源重试
    use_backup_map()
except Exception:
    # 其他未知问题：记录并上报
    log_and_alert()
```

1. 先尝试把 G 转换成一个加权有向图。
2. 如果转换过程中发生的是 GraphConversionError，说明是“图转换失败”这种已知问题，那就使用备用地图数据。
3. 如果发生的是其他任何未知错误，就记录错误并发出警报。


# What is the Exception class?

Exception is the parent class of nearly all errors in Python. Inherit from it to define your own error type, and you automatically get the ability to be raised, caught by try/except, and printed as a traceback.


## The exception family tree

```
BaseException          ← top (rarely touched directly)
 ├── SystemExit        ← from sys.exit()
 ├── KeyboardInterrupt ← Ctrl+C
 └── Exception         ← all everyday errors live here
      ├── ValueError
      ├── KeyError
      ├── TypeError
      └── ...
```

Inherit from Exception, NOT BaseException — so except Exception: won't accidentally swallow Ctrl+C or program exit.


## Why inheriting is the "entry ticket"
You can't raise an object that isn't in the exception family — Python rejects it with TypeError: exceptions must derive from BaseException. Inheriting from Exception gives you, for free:

1. message storage
2. traceback generation
3. catchability via except


### Standard custom exception

```python
class MyCustomError(Exception):
    """Describe when this happens."""
    pass
```

pass means "the parent's behavior is enough." Good for 99% of cases.


### With custom fields

```python
class LocationNotFoundError(Exception):
    def __init__(self, name, suggestions=None):
        super().__init__(f"Can't find '{name}'")   # parent stores the message
        self.name = name
        self.suggestions = suggestions or []
```
super().__init__(...) calls the parent Exception's initializer — this is how you "inherit those built-in abilities" in practice.
super() 指的是父类，也就是 Exception。所以这句的意思是：
调用父类 Exception 的初始化方法，把错误信息存进去。


当你这样写的时候：
```python
raise LocationNotFoundError("Shanghai")
```
Python 会自动调用：
```python
__init__(self, name="Shanghai", suggestions=None)
```

suggestions表示推荐的相似地点，比如：["Shanghai, China", "Shanghai Road"]
默认是 None，意思是你可以不传建议。
