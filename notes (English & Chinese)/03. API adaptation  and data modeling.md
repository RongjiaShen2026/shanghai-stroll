
# 1. API 适配 / 数据建模

工业级项目里 60-80% 的代码本质上都是"适配 / 转换"代码——只是被人冠以各种花哨的名字。新手以为编程主要在写"算法"，但实际上工业项目里真正的"算法"代码很少，大部分代码都在搬运、清洗、转换数据。软件工程里有个经典书叫《设计模式》（GoF 四人组写的，1994 年），列了 23 个常用模式。其中第二个就是 Adapter Pattern（适配器模式）：

Adapter Pattern：当两个系统接口不兼容时，写一个中间层把一边翻译成另一边能理解的格式。

业内的一句流行话

"Software engineering is mostly plumbing."
（软件工程主要是水管工活。）

意思是：把数据从一个地方"通过管道"运到另一个地方，中途做点过滤和转换。真正"发明新东西"的代码极少。

## 1. 实际工业项目中的"适配/转换"场景

1. Web 后端：API 接口

电商后端的典型流程：
```
用户在前端点击"下单"
   ↓
前端发送 JSON 请求 (前端的数据格式)
   ↓
后端 API 层: 转换 JSON → 内部业务对象  ← 适配器!
   ↓
业务逻辑层: 处理订单 (用内部对象)
   ↓
数据库层: 转换内部对象 → SQL 表行    ← 适配器!
   ↓
存进 MySQL
```
两个适配器，各占大量代码。一个典型的 Java/Python 后端项目里，"DTO（数据传输对象）转换"代码可能占总代码量的 20-30%。

2. 数据工程 / ETL

ETL = Extract（提取）、Transform（转换）、Load（加载）。整个领域的名字里就有"Transform"——整个行业就是在做适配。

```
源系统 A (Oracle 数据库)  
   ↓ 提取
原始数据
   ↓ 转换 (改字段名、合并表、清洗格式)  ← 适配!
干净数据
   ↓ 加载
目标系统 B (Snowflake 数据仓库)
```
整个数据工程师的工作 80% 是在写 Transform 代码。


3. 机器学习

```
原始数据 (CSV, 数据库, 图片, 日志)
   ↓ 适配: 清洗、归一化、特征工程       ← 大量适配代码
模型可以吃的张量 (numpy / torch tensor)
   ↓ 训练
模型
   ↓ 适配: 输出概率 → 业务可用的标签    ← 又是适配
业务系统能用的结果
```

在工业级 ML 项目里，数据预处理代码通常比模型代码多 10 倍。Andrew Ng 一直在讲 "data-centric AI"——意思就是搞数据/做适配比调模型重要。

4. 微服务架构

现代大公司的后端通常拆成几十上百个微服务，每个服务用不同语言、不同数据格式：
```
订单服务 (Java, 用 protobuf)
   ↕ 互相调用时要做格式转换  ← 适配
支付服务 (Go, 用 JSON)
   ↕ 适配
库存服务 (Python, 用 GraphQL)
```
每两个服务之间都有一层适配代码。这种代码在大公司里数量惊人。

5. 第三方集成
任何时候你的系统要接入外部 API（Stripe 支付、AWS S3、Google Maps、OpenAI），都要写适配：

```
你的系统格式
   ↕ 适配
Stripe API 格式
```
每接一个新服务就多一层适配代码。


6. 库之间的桥接

```
OSMnx 下载的数据 (NetworkX 格式)
   ↕ 你写的适配器
你的 Dijkstra 实现 (WeightedDigraph 格式)

```

跟工业项目里"接入第三方库"是同一类任务。


在不同的工业语境下，"适配器"有不同的名字：

|工业用语|本质|
|---|---|
|DTO (Data Transfer Object) 转换|后端 API 层适配|
|Serializer / Deserializer|对象 ↔ JSON/XML 适配|
|ORM (Object-Relational Mapping)|对象 ↔ 数据库表适配（SQLAlchemy、Django ORM、Hibernate）|
|Wrapper / 包装类|给老接口套个新接口|
|Bridge / 桥接|跨语言、跨协议适配|
|Connector / Driver|数据库驱动、API connector|
|Mapper|字段映射，常见于 ETL|
|Codec|编解码（音视频、网络协议）|
|Marshaller / Unmarshaller|内存对象 ↔ 字节流|
|Pipeline 中的 Transform 步骤|数据处理流水线里的转换节点|
|Middleware|请求/响应链路中的拦截转换|


为什么工业项目里这种代码这么多？

因为现实世界就是碎片化的：

1. 不同系统用不同的数据格式：MySQL、MongoDB、Redis、Elasticsearch……每个用法都不一样
2. 不同语言有不同的对象模型：Java 的 POJO、Python 的 dict、JavaScript 的 object……
3. 不同团队定义不同的术语：A 团队叫 user_id，B 团队叫 userId，C 团队叫 uid
4. 历史遗留系统不能动：30 年前的银行系统还在跑，新系统要接它就得适配
5. 第三方接口不受你控制：Google API 改了字段名，你得跟着改适配代码

所以整个软件行业一大半的工作就是缝合这些碎片。

"脏活累活"但极其重要

适配代码常被新手看不起——觉得没技术含量。但真相是：
1. 它最容易出 bug：边界情况、空值、类型转换错误，bug 都藏在这里
2. 它影响整个系统的稳定性：适配层一崩，下游全崩
3. 它是性能瓶颈常见来源：大量数据要转换时，慢一点就出大问题
4. 它考验真功夫：能写出干净、可维护、高性能适配代码的人，才是好工程师


## 2. 思考API 适配 / 数据建模的方式：

"看清两边，搭桥过河。"

1. 两边：搞清楚输入的数据结构 + 输出的数据结构
2. 搭桥：写代码把一边翻译成另一边

一个可复用的"思考模板"

你以后做任何"用 A 库的数据喂给 B 库"的任务时，都可以照这个模板想：

```
┌─────────────────────────────────────────┐
│  1. INPUT  (输入是什么?)                │
│     - 整体类型?                          │
│     - 元素类型?                          │
│     - 怎么遍历/访问?                     │
│     - 关键属性在哪?                      │
├─────────────────────────────────────────┤
│  2. OUTPUT (输出要什么?)                │
│     - 期望的整体类型?                    │
│     - 期望的元素类型?                    │
│     - 接口/方法签名?                     │
├─────────────────────────────────────────┤
│  3. GAP    (差距在哪?)                  │
│     - 类型不匹配?                        │
│     - 命名不一样?                        │
│     - 结构不同(嵌套 vs 扁平)?            │
│     - 需要保留对应关系?                  │
├─────────────────────────────────────────┤
│  4. BRIDGE (怎么搭桥?)                  │
│     - 遍历输入                           │
│     - 一边读一边转换                     │
│     - 必要时保存映射关系                 │
└─────────────────────────────────────────┘
```

对于转换器类的问题：思考这4个parts：

1. 我手上有什么数据？类型？结构？
2. 我要的输出是什么？类型？结构？
3. 中间的差距在哪？
4. 我打算怎么遍历输入、生成输出？需不需要存映射？


# 2. 具体解析

```python
"""
Graph adapter: converts NetworkX MultiDiGraph (from OSMnx) into our
internal WeightedDigraph representation used by the Dijkstra solver.
"""

from __future__ import annotations

import logging
import time
from typing import Tuple, Dict, Any, Hashable

import networkx as nx

from graph import WeightedDigraph, Node  # 你自己的模块

logger = logging.getLogger(__name__)


class GraphConversionError(Exception):
    """Raised when a NetworkX graph cannot be converted to a WeightedDigraph."""


def nx_to_weighted_digraph(
    G: nx.MultiDiGraph,
    *,
    weight_attr: str = "length",
    strict: bool = True,
) -> Tuple[WeightedDigraph, Dict[Hashable, Node]]:
    """Convert a NetworkX MultiDiGraph (from OSMnx) into a WeightedDigraph.

    Each OSM node becomes a Node in the output graph. For each parallel edge
    between two OSM nodes, only the edge with the smallest `weight_attr` is
    kept (since the shortest-path solver only cares about the minimum).

    Args:
        G: A NetworkX MultiDiGraph, typically produced by `osmnx.graph_from_*`.
            Nodes are expected to be OSM IDs (int). Edges must carry a numeric
            `weight_attr` (default: "length" in meters).
        weight_attr: Name of the edge attribute to use as edge weight.
        strict: If True, raise on missing/invalid weights. If False, skip
            offending edges and log a warning.

    Returns:
        A tuple ``(wg, osmid_to_node)`` where:
            - ``wg`` is the converted WeightedDigraph.
            - ``osmid_to_node`` maps original OSM IDs to the new Node objects,
              so callers can translate paths back to OSM IDs afterward.

    Raises:
        GraphConversionError: If the input graph is empty, or if `strict=True`
            and any edge is missing/has invalid `weight_attr`.
    """
    # ---- 1. Validate input ----
    if G is None:
        raise GraphConversionError("Input graph is None.")
    if G.number_of_nodes() == 0:
        raise GraphConversionError("Input graph has no nodes.")

    logger.info(
        "Converting graph: %d nodes, %d edges (weight_attr=%r, strict=%s)",
        G.number_of_nodes(), G.number_of_edges(), weight_attr, strict,
    )
    t0 = time.perf_counter()

    wg = WeightedDigraph()
    osmid_to_node: Dict[Hashable, Node] = {}

    # ---- 2. Convert nodes ----
    for osmid in G.nodes:
        node = Node(str(osmid))
        wg.addNode(node)
        osmid_to_node[osmid] = node

    # ---- 3. Convert edges (collapse parallel edges by minimum weight) ----
    skipped = 0
    for u, v, key, data in G.edges(keys=True, data=True):
        weight = data.get(weight_attr)

        if weight is None or not isinstance(weight, (int, float)) or weight < 0:
            msg = (
                f"Edge ({u}, {v}, key={key}) has invalid {weight_attr}={weight!r}"
            )
            if strict:
                raise GraphConversionError(msg)
            logger.warning("%s — skipping", msg)
            skipped += 1
            continue

        src = osmid_to_node[u]
        dst = osmid_to_node[v]
        # If multiple parallel edges exist, keep the shortest.
        existing = wg.getEdgeWeight(src, dst) if wg.hasEdge(src, dst) else None
        if existing is None or weight < existing:
            wg.addEdge(src, dst, float(weight))

    elapsed = time.perf_counter() - t0
    logger.info(
        "Conversion complete in %.3fs (%d edges skipped).", elapsed, skipped,
    )

    return wg, osmid_to_node
```

## 1. 
```python
from __future__ import annotations 
```
一个问题场景

假设你在写一个链表（Linked List）的类。链表的每个节点都有一个 next 属性，指向下一个同类型的节点。你很自然地会这么写：

```python
class Node:
    def __init__(self, value: int, next: Node) -> None:
        self.value = value
        self.next = next
```
这段代码会直接报错：

```python
NameError: name 'Node' is not defined
```

为什么会报错？

因为 Python 在执行 class Node: 这一行时，是从上往下逐行解析的。当它读到 def __init__(self, value: int, next: Node) 这一行时：

Node 这个类还没定义完, 此时 Python 内存里还没有 Node 这个名字,但你已经在用 Node 当类型注解了。

**from __future__ import annotations 干了什么**

它让 Python 自动把所有类型注解都当成字符串处理，你不用再手动加引号了。

```python
from __future__ import annotations

class Node:
    def __init__(self, value: int, next: Node) -> None:  # ← 不用引号，也不报错
        self.value = value
        self.next = next
```
Python 看到 next: Node 时，不会立刻去查 Node 是什么，而是默默把它存成字符串 "Node"，等真正需要用的时候（比如类型检查工具来分析时）再去查。

**这就是所谓的"延迟求值（lazy evaluation）"或者"惰性求值"。**


### 为什么叫 __future__？

因为这是一个预览未来 Python 行为的开关机制。

Python 团队有时候会想改某个行为，但又怕破坏现有代码，所以就先做成"可选开关"。你想用新行为，就 from __future__ import xxx 开启它；不想用，就维持旧行为。

"注解默认变成字符串"这个特性，Python 原本计划在 3.10 版本变成默认行为，但后来推迟了。所以现在还得显式写 from __future__ import annotations 才能启用。


其他注释：

-> None 是 Python 的返回值类型注解，专门用来告诉别人"这个函数返回什么类型"。

基本语法

Python 函数的完整类型注解长这样：

```python
def 函数名(参数: 参数类型) -> 返回值类型:
    ...
```
注意那个 ->（一个减号加一个大于号），它就像一个箭头，指向函数的"输出"。

-> None 是什么意思？

字面意思就是：这个函数不返回任何有意义的东西。
None 是 Python 里表示"空"、"什么都没有"的特殊值（类似其他语言的 null 或 void）。

```python
def greet(name: str) -> None:
    print(f"Hello, {name}!")
    # 没有 return 语句
```
这里函数只是打印一句话，没有 return，所以返回值是 None。


### 为什么 __init__ 总是 -> None？
因为 __init__ 是构造函数，它的工作是"初始化对象"，不是"创建并返回对象"。
很多人会误以为 __init__ 返回了新创建的对象，其实不是——对象是 Python 在背后帮你创建的，__init__ 只负责给这个已经创建好的对象设置初始属性。

注意：__init__ 不创建对象，它只是在对象被创建出来之后，立刻给这个对象贴上各种属性标签，让它从一个空壳变成一个"能用的、有内容的"对象。

## 6. 
```python
def nx_to_weighted_digraph(
        G: nx.MultiDigraph,
        *,
        weight_attr: str = 'length',
        strict: bool = True,
) -> Tuple[WeightedDigraph, Dict[Hashable, Node]]:
```

1. strict: bool = True

```
strict: bool = True
  ↑      ↑     ↑
参数名  类型注解  默认值
```
strict —— 参数名

: bool —— 类型注解，告诉别人"这个参数应该是布尔值（True / False）"

= True —— 默认值，意思是"如果调用时不传这个参数，就默认用 True"

strict 控制的是"遇到坏数据要不要报错"：

```python
if strict:
    raise GraphConversionError(msg)   # strict=True：直接报错
logger.warning("%s — skipping", msg)  # strict=False：只警告，跳过这条边
skipped += 1
```

1. strict=True（默认）：严格模式，遇到有问题的边就直接抛异常停下来
2. strict=False：宽容模式，遇到问题边就跳过、记个警告，继续干活

默认设为 True 是因为作者认为"默认情况下应该严格，发现问题就该报错，别偷偷藏着 bug"。


2. 
* : 它不是参数，而是一个分隔符。

它的作用是：强制它后面的所有参数，调用时必须用"关键字"形式传，不能用"位置"形式传。这样设计是为了可读性，weight_attr="time" 比 "time" 可读性强太多了。这是 Python 里一种防呆设计，很多高质量库都这么写。

什么叫"位置"和"关键字"？

```python
def func(a, b):
    ...

func(1, 2)        # 位置传参：靠顺序，1 给 a，2 给 b
func(a=1, b=2)    # 关键字传参：靠名字，明确写出谁是谁
```

* 划了一条线
```python
def nx_to_weighted_digraph(
    G: nx.MultiDiGraph,       # ← 在 * 前面：可以位置传，也可以关键字传
    *,                        # ← 这条分割线
    weight_attr: str = "length",  # ← 在 * 后面：只能关键字传
    strict: bool = True,          # ← 在 * 后面：只能关键字传
):
```

正确的调用方式：

```python
# G 位置传，后面的用关键字传
nx_to_weighted_digraph(my_graph, weight_attr="time", strict=False)

# G 也用关键字传也行
nx_to_weighted_digraph(G=my_graph, weight_attr="time", strict=False)

# 后面的参数不传，用默认值
nx_to_weighted_digraph(my_graph)
```

## 7. 
```python
logger.info(
    "Converting graph: %d nodes, %d edges (weight_attr=%r, strict=%s)",
    G.number_of_nodes(),    # 填到第一个 %d
    G.number_of_edges(),    # 填到第二个 %d
    weight_attr,            # 填到 %r
    strict,                 # 填到 %s
)
```
这行代码是在用 logging 记录一条信息日志。

这是老式的 C 风格格式化字符串：

|占位符|含义|例子|
|---|---|---|
|%d|整数（decimal）|42|
|%s|字符串（string，也接受其他类型）|True|
|%r|repr形式（带引号显示）|'length'|
|%f|浮点数|3.14|

假设 G 有 1000 个节点、2000 条边，weight_attr="length"，strict=True，最终输出：
```
Converting graph: 1000 nodes, 2000 edges (weight_attr='length', strict=True)
```

注意 %r 和 %s 的区别：
1. %s 显示 length（不带引号）
2. %r 显示 'length'（带引号，让你一眼看出这是个字符串）

### 为什么不用 f-string (f-string明明更简洁)？
```python
logger.info(f"Converting graph: {G.number_of_nodes()} nodes, ...")
```
这是个日志领域的最佳实践问题：

```python
# ❌ 不推荐
logger.debug(f"Processing {expensive_function()}")

# ✅ 推荐
logger.debug("Processing %s", expensive_function())
```

区别在于：

### 1. f-string：字符串立刻被拼接，即使日志级别被设成 WARNING（DEBUG 信息根本不会输出），expensive_function() 还是会被调用
### 2. % 风格 + 多参数：字符串只在真的要输出时才拼接，如果日志级别屏蔽了这条日志，参数和占位符根本不会被处理，省性能

对于 info 这种小开销日志区别不大，但养成习惯总是好的。所以专业代码里，logger.xxx 几乎都用 "...", arg1, arg2 这种写法。

## 8.
```python
osmid_to_node: Dict[Hashable, Node]={}
```
这种写法叫带类型注解的变量赋值（annotated assignment），是 Python 3.6 引入的语法。

可以理解成三件事同时发生：
1. 声明变量：osmid_to_node
2. 标注类型：说明它是一个"键是 Hashable、值是 Node"的字典
3. 赋初始值：给它赋值一个空字典 {}

### 空字典需要标注，非空字典不用
```python
# 非空字典：类型可以"自动猜"
prices = {"apple": 5, "banana": 3}
# 工具一看就知道：键是 str，值是 int

# 空字典：类型猜不出来
data = {}
# 工具懵：键是啥？值是啥？不知道
```

空字典 {} 里啥都没有，类型检查器（mypy、IDE）没办法推断出它装的是什么类型，只能笼统地认为是 Dict[Any, Any]（什么都行）。

加上类型注解后，工具就知道了：
```python
data: Dict[str, int] = {}
# 工具：明白了，以后只能往里放 str→int 的键值对
```

之后如果你不小心写错：
```python
data["apple"] = "five"   # 值应该是 int，结果你放了 str
```
IDE 或 mypy 会立刻警告你，避免 bug。

```python
if weight is None or not isinstance(weight, (int, float)) or weight < 0:
            msg = (f"Edge ({u},{v}, key={key}) has invalid {weight_attr}={weight!r}"
                   )
            
            if strict:
                raise GraphConversionError(msg)
            
            logger.warning("%s - skipping", msg)
            skipped += 1
            continue
```

## 9.1 第一行：条件判断
```python
if weight is None or not isinstance(weight, (int, float)) or weight < 0:
```
这是个三段式的"或"判断，任何一个成立就进入 if 块。三种"坏数据"的情况：

|条件|翻译|例子|
|---|---|---|
|weight is None|这条边没有 length 属性|data.get('length') 返回 None|
|not isinstance(weight, (int, float))|拿到的不是数字|居然是个字符串 '87.4'，或者别的怪东西|
|weight < 0|是负数|Dijkstra 不接受负权重，会出 bug|

几个语法点：

1. is None —— 检查是不是 None 的标准写法。不要写 weight == None（虽然能跑，但不规范）。
2. isinstance(x, (int, float)) —— 检查 x 是不是这两种类型之一。元组 (int, float) 表示"或"。
3. not isinstance(...) —— "不是"，加 not 取反。
4. 三个条件用 or 串起来 —— 只要中一个就算"坏边"。


### isinstance 是 Python 里最常用的内置函数之一。

**isinstance(x, T) 检查 x 是不是 T 这个类型的实例，返回 True 或 False。**

**读法：isinstance(x, T) = "is x an instance of T?"——"x 是 T 的实例吗？"**

**not isinstance(...) —— "不是"，加 not 取反。**

```python
isinstance(5, int)         # True   — 5 是 int 类型
isinstance(5, float)       # False  — 5 不是 float
isinstance(3.14, float)    # True
isinstance("hello", str)   # True
isinstance([1, 2], list)   # True
isinstance({'a': 1}, dict) # True
```

一次检查多种类型

第二个参数可以是一个元组，表示"满足其中任意一种就算"：
```python
isinstance(5, (int, float))     # True  — 5 是 int 或 float
isinstance(3.14, (int, float))  # True  — 3.14 满足 float
isinstance("5", (int, float))   # False — "5" 是字符串，两个都不是
```

## 9.2 第二行：构造错误消息
```python
msg = (f"Edge ({u},{v}, key={key}) has invalid {weight_attr}={weight!r}"
       )
```
用 f-string 拼一个描述性的错误消息，方便人看。

几个细节：
1. f"..." —— f-string，里面的 {变量} 会被替换成变量的值
2. {weight!r} —— !r 表示用 repr() 形式打印（带引号、显示类型）。比如 weight = 'abc'，普通打印是 abc，!r 打印是 'abc'，一眼能看出它是字符串而不是数字。这是调试时的小技巧。

下面这三行是等价的：
```python
print(f"{weight!r}")     # f-string 的 !r 修饰符
print(repr(weight))      # 直接调用 repr() 函数
print(f"{repr(weight)}") # 嵌套调用（啰嗦版）
```

同类还有：
```python
{x!s}    # 调用 str(x)（一般和默认一样）
{x!r}    # 调用 repr(x)
{x!a}    # 调用 ascii(x)（处理 Unicode）
```

3. 外层那个 (...) 括号 —— 只是为了让长字符串可以换行写，没有别的意义

## 9.3 第三到第五行：根据模式决定怎么处理
```python
if strict:
    raise GraphConversionError(msg)
```

如果调用者传了 strict=True（"严格模式"），那遇到坏数据直接抛异常，让程序停下来。

## 9.4 第六到第八行：宽容模式
```python
logger.warning("%s - skipping", msg)
skipped += 1
continue
```
如果不是严格模式（默认情况），就：

1. 打印一条警告（不是错误）—— logger.warning(...) 是 Python 标准的日志输出，比 print 专业。%s - skipping 是模板，%s 会被后面的 msg 填进去。
2. skipped += 1 —— 计数器加 1，记录跳过了多少条坏边。函数末尾大概会汇报"一共跳过了 N 条"。
3. continue —— 跳过当前这次循环的剩余部分，直接进入下一条边。不会执行后面创建 WeightedEdge 的代码。









# API adaptation and data modeling

The Core Principle: "See Both Sides, Build the Bridge"
When connecting two systems or libraries, the work breaks down into two parts:

1. Both sides: Understand the input data structure and the output data structure
2. Bridge: Write code that translates one into the other


## A Reusable Mental Template
Whenever you face a task like "feed data from library A into library B," walk through these four questions:

1. INPUT — What do I have?

What's the overall type?

What's the element type?

How do I iterate or access it?

Where are the key attributes I need?

2. OUTPUT — What does the target expect?

What's the expected overall type?

What's the expected element type?

What's the interface or method signature?

3. GAP — Where's the mismatch?

Type mismatch?

Different naming conventions?

Different structure (nested vs. flat)?

Do I need to preserve a mapping between the two?

4. BRIDGE — How do I build the translation?

Iterate over the input

Transform as I go

Save the mapping if I need to translate back later


## What is isinstance?

isinstance(x, T) checks whether x is an instance of type T. Returns True or False.

```python
isinstance(5, int)                # True
isinstance("hello", str)          # True
isinstance(5, (int, float))       # True — tuple means "any of these"
```

Why you need it?

Python is dynamically typed — variables can hold anything. isinstance lets you verify what you actually got, especially for:

1. External data (API responses, file contents)
2. Function arguments
3. Branching logic based on type

isinstance vs type(x) ==

```python
type(5) == int            # True
isinstance(5, int)        # True
isinstance(True, int)     # True ★ — bool is a subclass of int
type(True) == int         # False
```
**Use isinstance, not type() ==. It respects inheritance and supports checking multiple types via tuples.**
