```python
"""
Graph adapter: converts NetworkX MultiDiGraph (from OSMnx) into our
internal WeightedDigraph representation used by the Dijkstra solver.
"""

from __future__ import annotations

import logging
import time
from typing import Tuple, Dict, Any, Hashable

import networkx as nx

from graph import WeightedDigraph, Node  # 你自己的模块

logger = logging.getLogger(__name__)


class GraphConversionError(Exception):
    """Raised when a NetworkX graph cannot be converted to a WeightedDigraph."""


def nx_to_weighted_digraph(
    G: nx.MultiDiGraph,
    *,
    weight_attr: str = "length",
    strict: bool = True,
) -> Tuple[WeightedDigraph, Dict[Hashable, Node]]:
    """Convert a NetworkX MultiDiGraph (from OSMnx) into a WeightedDigraph.

    Each OSM node becomes a Node in the output graph. For each parallel edge
    between two OSM nodes, only the edge with the smallest `weight_attr` is
    kept (since the shortest-path solver only cares about the minimum).

    Args:
        G: A NetworkX MultiDiGraph, typically produced by `osmnx.graph_from_*`.
            Nodes are expected to be OSM IDs (int). Edges must carry a numeric
            `weight_attr` (default: "length" in meters).
        weight_attr: Name of the edge attribute to use as edge weight.
        strict: If True, raise on missing/invalid weights. If False, skip
            offending edges and log a warning.

    Returns:
        A tuple ``(wg, osmid_to_node)`` where:
            - ``wg`` is the converted WeightedDigraph.
            - ``osmid_to_node`` maps original OSM IDs to the new Node objects,
              so callers can translate paths back to OSM IDs afterward.

    Raises:
        GraphConversionError: If the input graph is empty, or if `strict=True`
            and any edge is missing/has invalid `weight_attr`.
    """
    # ---- 1. Validate input ----
    if G is None:
        raise GraphConversionError("Input graph is None.")
    if G.number_of_nodes() == 0:
        raise GraphConversionError("Input graph has no nodes.")

    logger.info(
        "Converting graph: %d nodes, %d edges (weight_attr=%r, strict=%s)",
        G.number_of_nodes(), G.number_of_edges(), weight_attr, strict,
    )
    t0 = time.perf_counter()

    wg = WeightedDigraph()
    osmid_to_node: Dict[Hashable, Node] = {}

    # ---- 2. Convert nodes ----
    for osmid in G.nodes:
        node = Node(str(osmid))
        wg.addNode(node)
        osmid_to_node[osmid] = node

    # ---- 3. Convert edges (collapse parallel edges by minimum weight) ----
    skipped = 0
    for u, v, key, data in G.edges(keys=True, data=True):
        weight = data.get(weight_attr)

        if weight is None or not isinstance(weight, (int, float)) or weight < 0:
            msg = (
                f"Edge ({u}, {v}, key={key}) has invalid {weight_attr}={weight!r}"
            )
            if strict:
                raise GraphConversionError(msg)
            logger.warning("%s — skipping", msg)
            skipped += 1
            continue

        src = osmid_to_node[u]
        dst = osmid_to_node[v]
        # If multiple parallel edges exist, keep the shortest.
        existing = wg.getEdgeWeight(src, dst) if wg.hasEdge(src, dst) else None
        if existing is None or weight < existing:
            wg.addEdge(src, dst, float(weight))

    elapsed = time.perf_counter() - t0
    logger.info(
        "Conversion complete in %.3fs (%d edges skipped).", elapsed, skipped,
    )

    return wg, osmid_to_node
```


### 1. 模块顶部的 docstring 和 import 分组
改动：加了模块级 docstring；按 PEP 8 把 import 分成标准库 / 第三方 / 本地三组。
为什么：工业代码读的人远远多于写的人。第一眼看到的就是 docstring 和 import——它们立刻告诉读者"这个文件是干什么的、依赖什么"。


"I separate imports into stdlib, third-party, and local groups because it's the PEP 8 convention and makes dependencies obvious at a glance."


### 2. 自定义异常类 GraphConversionError
定义了一个专属异常，而不是直接 raise ValueError。
为什么：调用方可以精准捕获这一类错误——

```python
try:
    wg, _ = nx_to_weighted_digraph(G)
except GraphConversionError as e:
    # 只处理图转换错误,其他异常正常往上抛
```
如果你用通用 ValueError，调用方就分不清"是转换出错"还是"别的地方出错"。

"I define a domain-specific exception so callers can catch just conversion errors without accidentally swallowing unrelated bugs. It's a small thing but it really matters in larger codebases."


### 3. Type hints（类型注解）

函数签名加了类型，返回值也注明了。
这样的话，IDE 能给你智能提示，mypy 这类工具能静态检查类型错误
最重要的是：类型注解是给读者的文档——读者看一眼就知道这个函数吃什么、吐什么

"Type hints aren't enforced at runtime, but they're effectively executable documentation. They make the function's contract explicit, and most modern Python teams treat them as standard."


### 4. Keyword-only 参数（* 之后的参数）

weight_attr 和 strict 必须通过关键字传递，不能位置传。

```python
nx_to_weighted_digraph(G, weight_attr="length")  # ✅
nx_to_weighted_digraph(G, "length")              # ❌
```

为什么：

避免调用方写出 f(G, "length", True) 这种没法读懂的代码。强制写成 f(G, weight_attr="length", strict=True)，意图非常清晰。

I force keyword-only arguments for booleans and config flags because positional booleans are a classic readability anti-pattern — f(G, True, False) tells you nothing.


### 5. 丰富的 docstring（Google 风格）
详细描述 Args / Returns / Raises。
为什么：这是 Python 工业代码的事实标准。Sphinx 等工具能直接从 docstring 生成文档。写代码 5 分钟，写好 docstring 10 分钟——这是正确的比例。


### 6. 输入校验（Validation）

检查 G is None、G.number_of_nodes() == 0。

为什么：永远假设你的函数会被错误地调用。早失败（fail fast）比让 bug 在程序深处冒出来好得多。

"Garbage in, garbage out — but at least crash loud and early."


"I validate inputs at the boundary of the function. If someone passes a malformed graph, I want a clear error here — not a cryptic KeyError 50 lines deep in Dijkstra."


### 7. 结构化日志（Logging）

用 logger = logging.getLogger(__name__)，记录开始、结束、跳过的边数。

为什么：

永远不要在工业代码里用 print——print 没法关、没法分级、没法集中收集

logger 可以配置成写到文件、发到监控系统、按级别过滤
出问题时日志是你唯一的线索


"I use the standard logging module instead of print because logs in production need to be configurable, filterable, and aggregatable. print is fine for scripts; logging is mandatory for anything that runs in production."


### 8. 性能计时
用 time.perf_counter() 测了一下耗时，记到日志里。

为什么：跑大图时 5 秒还是 50 秒区别巨大。在日志里留下耗时数据，性能回归一目了然。


### 9. 处理 MultiDiGraph 的平行边

用 G.edges(keys=True, data=True) 拿到所有平行边，对每对 (u, v) 只保留最短的。

为什么：这是你原代码里可能漏掉的真实 bug。

NetworkX 的 MultiDiGraph 允许 u→v 之间有多条平行边（比如同一条路有两条记录）。Dijkstra 只需要其中最短的——其他都是噪声。

"OSMnx returns a MultiDiGraph, which means there can be multiple parallel edges between the same pair of nodes. For shortest-path purposes, only the minimum-weight edge matters. I collapse parallel edges explicitly — this is the kind of detail that's easy to miss and causes subtle correctness bugs."

**一定要理解数据本身，不是机械地写转换代码。**


### 10. strict 模式开关

加了 strict 参数，控制遇到坏数据时是抛错还是跳过。

为什么：现实数据永远是脏的。有时候你想严格（开发时），有时候你想容忍（生产数据有 0.01% 坏值，跳过就行）。给调用方选择权。

"Real-world data is messy. I let the caller choose whether to be strict or lenient — strict mode for development and testing, lenient mode when running on production data where you'd rather skip a few bad edges than crash."


### 11. 边权重的类型与值检查

检查 weight 不是 None、是数字、非负。

为什么：
1. Dijkstra 要求非负权重——负权重会导致结果错误
2. None 或字符串会让后续运算崩溃
3. **在入口处把所有边界情况堵死——下游代码就可以放心假设权重是 valid 的**


### 12. 显式 float(weight) 转换
wg.addEdge(src, dst, float(weight))。

为什么：NetworkX 里 length 有时候是 int 有时候是 float。统一成 float 避免下游做混合类型运算时的微妙 bug。


还能怎么改进？

1. 单元测试（Unit tests）

```python
# 在另一个文件 test_adapter.py 里
def test_empty_graph_raises():
    with pytest.raises(GraphConversionError):
        nx_to_weighted_digraph(nx.MultiDiGraph())

def test_parallel_edges_keep_min():
    G = nx.MultiDiGraph()
    G.add_edge(1, 2, length=10.0)
    G.add_edge(1, 2, length=5.0)   # 平行边
    wg, _ = nx_to_weighted_digraph(G)
    assert wg.getEdgeWeight(...) == 5.0   # 保留了最短的
```

"I'd write unit tests covering the happy path, empty input, parallel edges, and invalid weights. The adapter layer is exactly where edge-case bugs hide, so it deserves test coverage."


2. Round-trip testing（往返测试）
把转换后的图再转回 NetworkX，确认节点/边没丢。

3. 可观测性（Metrics / observability）
生产环境可能要把"转换耗时"、"跳过边数"发到 Prometheus 这类监控系统。日志只是第一步。

4. 内存优化
如果图非常大（百万节点），可以考虑流式处理或者用 __slots__ 优化 Node 类。但没必要过早优化——先写对、再写快。


"When I rewrote this adapter to be production-ready, I focused on a few things beyond just making the code work:
1. First, robustness. I added input validation, a custom exception type so callers can handle conversion errors specifically, and a strict flag so the caller can choose between failing fast or tolerating bad data.
2. Second, observability. I replaced print with the logging module and added timing instrumentation, because in production the only thing you have when something goes wrong is your logs.
3. Third, correctness on the actual data. OSMnx returns a MultiDiGraph with parallel edges, which my original code didn't handle — I now explicitly collapse them to the minimum weight, which is what the Dijkstra solver actually needs.
4. Fourth, the interface. Type hints, keyword-only arguments for flags, and a thorough docstring — so the next person who reads this function understands the contract without having to read the implementation.
The core logic is still just a few loops. Most of what changed is what I'd call 'production wrapping' — but that wrapping is what separates code that works on your laptop from code that runs reliably for years."


工业级代码的核心是：

1. 对未来的读者负责（清晰的命名、注释、类型）
2. 对未来的自己负责（防御性编程、错误处理、日志）
3. 对系统的稳定负责（验证、测试、监控）
